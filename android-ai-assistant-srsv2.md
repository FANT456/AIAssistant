# 本地 AI 行程助手项目规划（精简版）

> 目标：把“自然语言行程”自动转换成标准 JSON，并写入飞书日历  
> 当前策略：**先做 Windows 本地版本，跑通文字 / 文件 → LLM → JSON → 飞书日历 的闭环**

---

# 🧭 一、项目核心目标（重新定义）

## 🎯 最终目标

构建一个可以：

* 理解自然语言中的日程信息
* 从文本或文件中提取结构化字段
* 输出统一标准 JSON
* 调用飞书日历 API 自动创建日程
* 记录处理历史，便于追踪和后续增强

的本地 AI 行程助手系统。

---

## ⚠️ 当前阶段重点

> ❗ 当前不是做“完整 AI 助手”，而是做：
>
> 👉 **一个稳定的“行程抽取 + 入历”核心链路**

---

# 🏗 二、整体技术路线

## 当前架构（Phase 1）

```
文字输入 / 文件输入
        ↓
本地 LLM（Ollama）理解内容
        ↓
输出标准 JSON
        ↓
调用飞书日历 API
        ↓
SQLite 存储处理记录
```

---

## 长期架构（目标形态）

```
客户端（Windows / Android）
        ↓
本地 AI 服务（LLM + 规则校验 + RAG）
        ↓
工具系统（日历 / 提醒 / 文件 / 邮件）
        ↓
外部系统（飞书 / Outlook / 本地设备）
```

---

# 🚀 三、开发阶段规划（Phases）

---

# 🥇 Phase 1：行程抽取与飞书入历（当前阶段）

## 🎯 目标

> 👉 跑通“输入 → 理解 → JSON → 写入日历 → 存储”的完整闭环

---

## ✅ 功能范围

### 1. 本地大模型运行

* 使用 Ollama
* 支持本地推理
* 不依赖云端 AI 模型服务

---

### 2. 输入方式支持

* 支持直接粘贴行程文字
* 支持读取本地文本文件内容
* 统一送入大模型理解

示例：

```
公司定于2026年3月23日，下午13:30召开第一季度安全生产会议。
地点在总部 3 号会议室，参会人员包括部门负责人和质量专员。
```

---

### 3. 标准 JSON 输出

大模型输出统一结构，例如：

```json
{
  "title": "第一季度安全生产会议",
  "start_time": "2026-03-23 13:30",
  "end_time": "2026-03-23 14:30",
  "timezone": "Asia/Shanghai",
  "location": "总部3号会议室",
  "attendees": ["部门负责人", "质量专员"],
  "description": "第一季度安全生产会议"
}
```

---

### 4. 飞书日历 API 调用

* 获取 tenant access token
* 创建飞书日历事件
* 返回创建结果（event_id / 错误信息）

---

### 5. SQLite 持久化

存储内容：

* 原始输入文本
* 模型返回 JSON
* 创建是否成功
* 飞书事件 ID / 错误信息
* 时间戳

---

## ❌ 不包含（必须控制范围）

* 多轮闲聊助手
* 复杂工具编排
* 语音 / 图片识别
* RAG（向量检索）
* Android 端
* 多人参会状态同步

---

## 🧩 技术选型

| 模块 | 技术 |
| --- | --- |
| LLM | Ollama |
| 语言 | Python |
| 存储 | SQLite |
| 日历 | 飞书 Calendar API |
| 通信 | HTTP |

---
